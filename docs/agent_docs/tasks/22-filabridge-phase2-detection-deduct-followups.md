### 22.1 ‚ÄĒ Cancel detection: no instant ACK + fast cancel‚Üírestart slips the 30s poll ‚ÄĒ ‚úÖ DONE 2026-06-13 (`54f271c`, unmerged)
**Buglist:** line 3. Two distinct gaps from a real XL cancel:

**Fix direction:** **same as 22.2** ‚ÄĒ a latched in-progress job that reaches IDLE without an OBSERVED STOPPED/FINISHED must NOT be silently dropped: download the now-unlocked file ‚Üí compute ‚Üí route to the cancel-review pipeline flagged "couldn't confirm what happened" (retain `progress:0.26` as the hint). **Mitigations to weigh:** shorter poll; a state-change fast-poll BURST in the daemon (mirror the Phase-0/L25 frontend fast-poll); or persist+compare last-seen state across ticks so a `PRINTING‚Üí(unseen STOPPED)‚ÜíIDLE` not matching a clean completion surfaces a review.

### 22.2 ‚ÄĒ Completion silently missed when the printer power-cycles before FCC observes FINISHED ‚ÄĒ ‚úÖ DONE 2026-06-13 (`54f271c`, unmerged ‚ÄĒ same change as 22.1)
**Buglist:** line 5. **ūüĒ¨ RESEARCH FINDINGS (2026-06-14 background workflow ‚ÄĒ HIGH confidence, source-traced to PrusaSlicer/Buddy code + PrusaLink OpenAPI; Derek's actual sliced file remains authoritative):**
- **The footer does NOT give per-segment grams for a single-extruder color print ‚ÄĒ my earlier "footer array in load order" assumption is WRONG.** Two distinct PrusaSlicer workflows behave differently:
  - **Color Change (layer-slider ‚Üí "Add color change", the common M600 workflow):** ONE configured extruder ‚Üí `; filament used [g] = 4.74` is a **SINGLE total**, no per-color breakdown at all. PrusaSlicer does not track grams per color here. ‚Üí per-segment grams MUST be computed by **integrating E-moves between boundaries**, not read.
  - **SEMM (Printer Settings ‚Üí Extruders=N + Single-Extruder-Multi-Material, assign objects/features to virtual extruders):** emits a comma ARRAY `; filament used [g] = 0.1, 0.3, 4.8, ‚Ä¶` ‚ÄĒ but **keyed by extruder-ID (0-based config order), NOT load/color-change order**, padded with `0.00` for unused extruders. So `array[i]` = extruder i (cross-ref the `T#` in the markers), never "the i-th M600".
  - `; total filament used [g] = ‚Ä¶` is ALWAYS a single grand-total (safe cross-check / equals the array sum).
- **Segmentation is GCODE-MARKER-DRIVEN, not footer-driven.** PrusaSlicer emits `;COLOR_CHANGE,T0,#RRGGBB` immediately before each planned `M600` ‚ÄĒ deterministic, carries the tool index + hex color, scannable for byte offset + the surrounding `;Z:`/layer. **Scan for `;COLOR_CHANGE` (not bare `M600` ‚ÄĒ users can override `color_change_gcode`).** Markers only exist AFTER `bgcode_decode.py` (the fleet is binary `.bgcode`). So (b)'s segmentation = decode ‚Üí scan `;COLOR_CHANGE` boundaries ‚Üí integrate extruded E within each segment ‚Üí grams ‚Üí attribute. This is an extension of the existing `parse_partial_filament_usage` E-integration, sliced at the markers ‚ÄĒ **no off-the-shelf community tool does single-extruder per-color accounting** (Spoolman/OctoPrint plugins attribute to one selected tool/spool).
- **Pause state:** M600 / color-change / **runout all show `state="ATTENTION"` on `/api/v1/status`** (the printer-state enum); `/api/v1/job` shows `state="PAUSED"` + filename + a FROZEN progress % at the pause point. The job-state enum has NO ATTENTION ‚ÄĒ so the monitor must read the **STATUS** endpoint to catch the pause and the **JOB** endpoint for the boundary progress %. No API field distinguishes M600 vs color-change vs runout (open enhancement #3059); only a *user* pause differs (PAUSED, not ATTENTION). Firmware-version caveat: older/MMU paths sometimes still report PRINTING on runout ‚ÄĒ defensive handling needed.
- **Design implication:** segmentation can be **purely gcode-derived** (the `;COLOR_CHANGE` byte/Z boundaries) ‚ÄĒ the swap-event hook (a) is then needed mainly for **attribution** (which spool was loaded per segment) + to confirm a swap actually happened. The swap_log load-order aligns with the `;COLOR_CHANGE` order (both temporal), so segment k ‚Üí spool-after-the-k-th-resume. For a clean Color-Change print this gives a full per-segment split with NO reliance on the footer array.
