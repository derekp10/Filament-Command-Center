## TL;DR — the Config system is built; this is the per-printer tail

L18 Phases 1–4 shipped 2026-06-01 (safe atomic writer, `GET/PUT /api/config`, the schema-driven Settings card, editable connection settings + masked `SCRAPER_API_KEY`, the guarded `printer_map` topology editor, import/export). What's left:

**N6 is a schema-type + renderer job, not backend plumbing.** The one real customer, `path_filament_g` (the runout→nozzle remnant, Group 22.3b), is **uniquely clean**: its reader `print_deduct._path_filament_g` **already accepts a `{printer_name: grams, "default": grams}` map** or a scalar ([print_deduct.py:921](../../../inventory-hub/print_deduct.py#L921)), pinned by [test_runout_auto_split.py:162-189](../../../inventory-hub/tests/test_runout_auto_split.py#L162), and [config.json.example:28](../../../config.json.example#L28) already documents the map form. So the **read side is done**; only three things are missing:
1. a schema **map field type** + per-cell coercion/validation ([config_schema.py](../../../inventory-hub/config_schema.py) is strictly flat-scalar today — a dict value falls to the `else` branch and gets `str()`'d, [:114](../../../inventory-hub/config_schema.py#L114));
2. a **frontend renderer** (the `inputFor()` if-ladder has no map branch — a dict renders into a single text box → silent corruption, [inv_settings.js:39](../../../inventory-hub/static/js/modules/inv_settings.js#L39));
3. a **printer-name source** for the renderer's rows.

**The Phase-3 `printer_map` editor is NOT the reusable machinery** — it's a *topology* map (`{LocationID: {printer_name, position}}`) with its own endpoint (`/api/printer_map`), validator (`_canonicalize_printer_map`), and store (locations.json Printer rows). Different shape, keys, and store from a per-printer *settings* map (`{printer_name: scalar}` in config.json). **The real reuse anchor is the per-printer CREDS grid** (`pcRender`/`pcRowHtml` [inv_settings.js:249-360](../../../inventory-hub/static/js/modules/inv_settings.js#L249)) — it already renders one row per printer Name in a CSS grid with per-cell masking + per-row save; a grams/offset grid is strictly *simpler* (no secret).

**Phase 5 co-location shares N6's structure.** N6's per-printer field wants a widget host co-located with the printer_map/creds block; Phase 5 introduces exactly a per-section host registry. Build the section-host once → it serves both. Phase 5 is UI-only (no endpoint change).

## ⚠️ Doc-vs-code drift (whoever builds this: trust the source, not the L18 design doc)
- No `WIDGETS` table exists — the renderer is an imperative `inputFor()` **if-ladder** ([inv_settings.js:39](../../../inventory-hub/static/js/modules/inv_settings.js#L39)); the design doc's "WIDGETS table" is aspirational.
- Schema serializer is `schema_for_ui()`, not `schema_as_json`; there is **no `CLIENT_PREF_KEYS`** ([config_schema.py:202](../../../inventory-hub/config_schema.py#L202), [:100](../../../inventory-hub/config_schema.py#L100)).
- Secret masking lives in the **route layer** (`api_get_config`/`api_config_export`), not the schema serializer.
- `printer_map` is **no longer written to config.json** — it persists on locations.json Printer rows post-L271-Phase-4; config:printer_map is a boot-priming seed nothing reads.
- Routes moved to `routes_config_attrs.py` / `routes_bindings.py` post-L316.

**Decisions with a safe default (stated, not blocking):**
- **Backing store = config.json** for per-printer *behavior* scalars (`path_filament_g`, future offsets — geometry constants); **creds stay on the Printer row** (`/api/printer_creds`). Do NOT relocate creds into a config secret-map — it's a done, masked, per-printer mechanism; duplicating it would break the exact-equality verify and split creds across two stores.
- **Schema type shape:** one generic `per_printer_map` type + a new `cell_type` `Field` attribute (reuse the scalar vocab: `float`/`int`) reusing `min`/`max` as per-cell bounds (path keeps `min=0`/`max=50` per cell). One renderer branch, one coerce branch dispatching on `cell_type`.
- **Register in `CONFIG_SCHEMA`** (so the field participates in `GET/PUT` validation + export/import) but **render via a dedicated per-printer widget host** co-located with the printer_map/creds block — the hybrid that also satisfies Phase 5.
- **Coercion keeps `config_schema.py` pure** (imports only `math`+`re`): validate each *cell value* by `cell_type` + range; **do NOT** validate map *keys* against live printer names (that would couple the pure schema to `locations_db` → cycle risk). The reader already tolerates unknown Names via `"default"`. The renderer supplies the name list.
- **On-disk shape:** read dual-shape (reader already does), but the **UI always writes a map with a `"default"` entry** so the on-disk form is uniform going forward — no migration script needed (an existing scalar just keeps working until the next save).

## ⚠️ Adversarial-review corrections (2026-07-07)

1. **The `per_printer_map` coerce branch + renderer MUST tolerate a bare scalar (HIGH).** `coerce_and_validate` is the write/validate path, hit on **import** ([routes_config_attrs.py:128](../../../inventory-hub/routes_config_attrs.py#L128) → `validate_payload`) and PUT. `config.json.example:29` ships `path_filament_g` as the scalar `0` and the schema default is `0`, so real configs + exported backups carry a scalar — a dict-only branch would **400 the whole import** (and `test_import_round_trips_redacted_export` exercises export→import). Also `GET /api/config` returns the raw scalar for an unset field, so on **first modal-open** for every upgrading install the widget receives a scalar. Require: the coerce branch **normalizes a bare int/float → `{"default": <coerced>}`** (mirroring the reader's scalar path), and the renderer **treats a scalar `values[key]` as the `default` cell**. Add a round-trip test importing a legacy scalar config + a GET-returns-scalar render test.
2. **Phase-5 host-id preserve-list is incomplete — add `#config-generated-settings` (MEDIUM).** It's the schema-Settings-card host (`inv_settings.js` `HOST_ID`) where N6's per-printer field renders, and `render()`/`renderConfigSettings` **early-return silently** if it's missing — the exact "silently break" the doc warns about. Also add the import controls `#cfgio-file` / `#cfgio-import-btn` ([modals_config.html:124-125](../../../inventory-hub/templates/components/modals_config.html#L124)). State the invariant as "every id in {`config-generated-settings`, `config-attrs-results`, `config-restore-field-order-results`, `config-printer-map`, `config-build-info`, `cfgio-file`, `cfgio-import-btn`} still resolves after `openConfigModal()`" and add a smoke assertion.
3. **The import confirm-diff must handle dict values (MEDIUM).** The diff builds rows via `str(cur) != str(new)` and renders `_esc(x.from/x.to)` — for a dict that's key-order-sensitive and shows `[object Object] → [object Object]` ([routes_config_attrs.py:141](../../../inventory-hub/routes_config_attrs.py#L141), [inv_settings.js:448](../../../inventory-hub/static/js/modules/inv_settings.js#L448)). Since Apply is gated on `diff.length`, the review UX is load-bearing. Require: compare `json.dumps(…, sort_keys=True)` and render via `JSON.stringify` for dict-valued fields; add a map-valued import-diff test. *(Note: the doc's "dict equality is key-order-insensitive" reassurance refers to `_verify_config_file`'s `parsed != expected_dict` — a different, genuinely order-insensitive path — NOT this `str()` diff.)*
4. **Blank per-printer cell = OMIT the key, not write `0` (LOW).** The reader inherits `"default"` only for an **absent** key ([print_deduct.py:934](../../../inventory-hub/print_deduct.py#L934)); writing `0` pins that printer to 0 g (un-settable later) and `float('')` on an empty cell would 400 the save. Require: the collect step **skips empty cells before coercion**; only the `default` cell is always written. Pin: a printer left blank resolves to the default grams.
5. **Printer-name source (verified OK — no change).** The name list is the **union** of `GET /api/printer_map` `printers`/`entries` printer_names **and `printer_creds` keys** (de-duped by Name) — exactly what `pcNames` does ([inv_settings.js:268-272](../../../inventory-hub/static/js/modules/inv_settings.js#L268)); this already captures a **toolhead-less** Printer row (e.g. dual-role CORE1) via the creds view. Phase 3 wording just echoes this so a builder skimming phases doesn't source from the toolhead-only list.

## Phased build plan

- **Phase 1 (optional, recommended) — Renderer refactor.** Turn `inputFor()`'s if-ladder + `save()`'s type-branches into a `WIDGETS = {type: {build, read}}` table (G4). Behavior-preserving; existing config E2E/round-trip tests stay green. Makes the map a clean new entry rather than a bolt-on.
- **Phase 2 — Backend map type.** Add `per_printer_map` + `cell_type` to the `Field` dataclass; a `coerce_and_validate` branch that **accepts a bare scalar → `{"default": x}`** (correction #1) and per-cell dispatches into the (factored-out) scalar coercer + applies `min`/`max`, **skipping empty cells** (correction #4); extend `validate_payload` + `schema_for_ui` + the export/import round-trip **including the `json.dumps(sort_keys=True)` diff fix** (correction #3). Fix the stale `Field.type` docstring. Pin with a `load(save(x)) == load(x)` round-trip, a **legacy-scalar import** test, a per-cell range-reject, and a blank-cell-inherits-default test.
- **Phase 3 — Per-printer renderer + name source.** A `WIDGETS['per_printer_map']` widget cloning `pcRowHtml`'s grid (one row per Name + a `default`/all-others row), sourcing names from the `GET /api/printer_map` fetch already made on modal open (de-dupe by Name — `build_printer_map_from_rows` yields one entry per *toolhead*). Collect rows → dict → PUT `/api/config` (the `pmSave` idiom). Migrate `path_filament_g`'s `Field` from `float` to `per_printer_map` (`cell_type='float'`, `min=0`, `max=50`).
- **Phase 4 (G3) — Action-tool co-location (L18 Phase 5).** A section-keyed host registry: add tool-bearing sections to `config_schema.SECTIONS`, have the renderer emit a host `<div>` per section, and refactor `configReconcileScan`/`configAttrsScan`/`configRestoreFieldOrder` to accept a passed host so the Filament-Attributes manager mounts beside its prefs. **UI-only — preserve every fixed host id** (correction #2): `#config-generated-settings` (the schema-Settings host — the most important, `render()` early-returns silently if missing), `#config-attrs-results`, `#config-restore-field-order-results`, `#config-printer-map`, `#config-build-info`, `#cfgio-file`, `#cfgio-import-btn` — add a smoke assertion each resolves after `openConfigModal()` or they silently break. Overlaps the config-modal portion of the standalone "Filament-attribute add broken in wizard" bug.
- **Phase 5 — Tests + polish.** Config round-trip byte-identity for the map; per-cell validation reject; the migrated `path_filament_g` still deducts correctly on a runout split (the `_path_filament_g` consumer + `test_runout_auto_split` stay green); visual check of the per-printer grid; add the new type to CLAUDE.md's write-surface notes if warranted.

## Risks to respect
- **Both ends must land together.** A map-typed field with no renderer branch renders the dict into a text box ([inv_settings.js:83](../../../inventory-hub/static/js/modules/inv_settings.js#L83)) and `save()` ships the stringified dict; the backend then `str()`s it ([config_schema.py:152](../../../inventory-hub/config_schema.py#L152)) — silent corruption. Ship the schema type + renderer in the same change.
- **Exact-equality verify** ([config_loader.py:289-303](../../../inventory-hub/config_loader.py#L289)) fails on non-deterministic coercion. Coerce cells deterministically (stable float coercion; dict equality is key-order-insensitive, so key order is safe). Add the round-trip test.
- **Enumeration drift** — reading `config.json:printer_map` keys is the natural but **wrong** instinct (vestigial seed post-L271-Phase-4). Enumerate from Printer rows / `GET /api/printer_map`.
- **De-dupe by Name** — `build_printer_map_from_rows` yields one entry *per toolhead* (XL-1..XL-5 all Name "XL"); a naive renderer shows 5 rows for one printer. Group by Name (as `printers`/`printer_creds` already do).
- **Rename orphaning** (G1=Name) — a printer rename silently orphans its map entry; low impact for grams (~2-4 g), higher for future offsets. Flag in field help; consider a rename-cascade later.
- **Phase 4 is UI-only** — the action-tool endpoints + their `window.*` wiring are load-bearing; keep it a DOM-reparent + host-wiring change and verify the Save/dirty flow still targets the right hosts.
- **`config.json` is a single-file bind-mount** (EBUSY `os.replace` → in-place fsync fallback, already handled in `_write_config_atomic`) — don't disturb that path ([[reference_fcc_config_json_bind_mount]]).

## Prior art / alignment
Directly continues L18 (Phases 1–4 shipped + a 30-agent post-ship audit). The per-printer renderer clones the already-working per-printer **creds** grid — the config modal *already* mixes the generic schema renderer with bespoke per-section editors, which is exactly the structure Phase 5 formalizes. This is a **small cluster** (one substantive item + one optional UI-reorg), not a large epic — right-sized as one working group once the forks are locked, and lower-risk than the Location-tree cluster (no prod data migration, no safety-critical matchers).
