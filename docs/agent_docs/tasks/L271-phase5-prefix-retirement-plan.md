## TL;DR — the buglist instruction is STALE; retirement = SPLIT, not delete

The literal buglist step ("delete `derive_parent_id_from_prefix`") is **unsafe as written** — the annotation predates the function growing load-bearing *non-migration runtime consumers* (matches the [[feedback_buglist_annotations_can_be_stale]] memory). Verified by grep 2026-07-07, `derive_parent_id_from_prefix` ([locations_db.py:349](../../../inventory-hub/locations_db.py#L349)) has **two distinct roles that must be separated**:

1. **Hierarchy derivation (RETIRE).** The `resolve_parent` fallback ([locations_db.py:381-382](../../../inventory-hub/locations_db.py#L381)) — the one the spec names — *plus* a second, unnamed twin inside `_parent_of` ([locations_db.py:427](../../../inventory-hub/locations_db.py#L427)) that `resolve_room` / `is_descendant` / `ancestors_of` all ride on. These fabricate phantom ancestor edges for any id that isn't a real row. This is the footgun Phase-5 kills.
2. **Flat location-STRING matching (KEEP — safety-critical).** Four callers in `spoolman_api.py` ([:1320, :1328, :1481, :1482](../../../inventory-hub/spoolman_api.py#L1320)) use it as a *first-segment string match*, deliberately kept flat by the 2026-06-04 adversarial review (fixes #1/#5/#9) so a room-level clear/delete reaches cart-rows but **NOT** a nested printer's actively-printing toolhead. The docstring at [:1298](../../../inventory-hub/spoolman_api.py#L1298) explicitly says "NOT a transitive `is_descendant` walk." Plus the `/api/locations` synthesizer ([routes_locations.py:70](../../../inventory-hub/routes_locations.py#L70)) calls it to stamp `parent_id` on never-persisted Spoolman-native virtual rows.

**A blanket delete would silently enlarge the destructive blast radius** (turn those flat matches into transitive subtree sweeps) — re-introducing the exact bug the 2026-06-04 review fixed, invisibly (no test failure unless pinned). So the retirement is: **introduce a renamed first-segment helper (`location_prefix()` / `first_segment()`) for the string-query + synthesizer callers, then delete `derive_parent_id_from_prefix` and its two *hierarchy* fallbacks.**

**Data is already safe.** All 55 on-disk rows carry a `parent_id` (0 missing, 0 true-orphans; committed file at the immediate-parent fixpoint — pinned by `test_parent_id_is_immediate_parent_fixpoint`, [test_locations_json_integrity.py:271](../../../inventory-hub/tests/test_locations_json_integrity.py#L271)). **10 rows legitimately parent onto rowless pseudo-prefixes** (PM/PJ/TST — [locations_db.py:390](../../../inventory-hub/locations_db.py#L390)), so "orphan" ≠ "parent not in existing LocationIDs"; the write-time check MUST whitelist `PSEUDO_ROOM_PREFIXES`.

## ⚠️ Adversarial-review corrections (2026-07-07) — READ FIRST; these SUPERSEDE the body below

A 5-lens adversarial review (2 lenses cross-confirmed the biggest finding) caught real defects in the first draft. Corrected scope:

1. **`location_prefix()` is the sanctioned first-segment helper at EVERY still-load-bearing site — not just the 4 matchers.** First-segment derivation is legitimately needed for **rowless ids** (pseudo-prefix PM/PJ/TST, Spoolman-native `location` strings, not-yet-created rows) that have no `parent_id` to trust. Drop the "mirrors `api_save_location`" framing (that's single-row; this is whole-list). Consider cascade-reparenting children on a parent-row delete ([routes_locations.py:387](../../../inventory-hub/routes_locations.py#L387) doesn't today) so orphans can't accrue.
6. **`location_prefix()` MUST return `None` for a no-dash id** (keep `derive_parent_id_from_prefix`'s guard verbatim). The 4 matchers are equality-masked (identity holds regardless), but the **synthesizer stamp** ([routes_locations.py:70](../../../inventory-hub/routes_locations.py#L70)) is NOT masked and depends on None-on-no-dash (else a dash-free Spoolman-native location self-parents). 7. **CI pins to add to the Phase-4 re-base list:** `test_l271_phase35_hierarchy.py` (`test_is_descendant_prefix_fallback_for_unknown_id`, `test_ancestors_of_prefix_fallback_for_unknown_id`), `test_l271_phase35_migration.py` (`test_immediate_parent_for_no_ancestor_falls_back_to_first_segment` + `_flat_tree` fixture), and `test_buffer_ejection_logic.py:150-155` (pins get_room_from_location fallback on rowless ids). With corrections #1–#3 these tests should stay GREEN (behavior preserved via `location_prefix()`), not be re-based — a bonus.
8. **Grep-gate:** add a 4th exclusion for the sanctioned helpers (`location_prefix`, `_immediate_parent_from_rows` which uses `'-'.join(parts[:i])`, not `[0]`); list `:941`/`:1185` as KEEP. **Net effect on scope:** Phase-5 shrinks from "delete prefix derivation" to "**rename** `derive_parent_id_from_prefix`→`location_prefix()` + re-point all load-bearing callers + add newly-changed-row write-validation + (with the add-redesign) make explicit-parent the UI default." ### 🚫 Hard invariant — NO FORCED RELABELING (Derek 2026-07-07)

Derek's Q4 note, elevated to a blocking constraint for the whole cluster: *"I want to avoid having to re-generate location labels as much as possible… I still have a lot [of labels] that miss the `LOC:` prefix out there. If we just straight-dump the existing lookup for this new id system, it will cause problems."* He already has a large legacy-relabeling backlog (spools too) and finds the reprint-nag actively demoralizing. So:

1. **Existing LocationIDs are IMMUTABLE through this work.** Retirement changes how the hierarchy is *resolved* (`parent_id`, not prefix), never the stored ids. Auto-generation (add-redesign S2) applies to **NEW rows only** — it never renames or re-mints an existing row.
2. **Scan resolution stays backward-compatible with EVERY existing label form.** The resolver today handles `LOC:<id>` ([logic.py:209](../../../inventory-hub/logic.py#L209)) and explicit legacy prefixes `LEGACY:`/`LEG:`/`OLD:` ([logic.py:216](../../../inventory-hub/logic.py#L216)) — plus Derek has many physical labels that carry the **bare id, no `LOC:` prefix** at all. Making `parent_id` authoritative must NOT narrow what resolves: a scanned bare/legacy location code must keep landing on its row exactly as today. **Edge-case-checklist item:** before shipping, verify the bare-token / legacy fallback paths still resolve a prefix-less location label under the new lookup (don't "straight-dump the existing lookup").
3. **Don't nag.** A relabel prompt fires ONLY when the printed identity actually changes (a deliberate rename) — never on a routine parent/type/field edit or on the retirement migration itself. Reprints stay opt-in and batchable, not a per-edit reminder of the backlog.

- **Barcode decoupling stays DEFERRED** (Derek 2026-06-03, reinforced 2026-07-07) — labels already read only stored fields + opaque `LOC:<id>` QR, so decoupling is *effectively done at the label layer*; the only things keeping the dash convention alive are the Auto-derive (P4) and the shelf-grouping migration (owned by the add-redesign Phase 4). No reprints, consistent with the invariant above.
- **The grep-gate CI test** asserts zero *hierarchy* `split('-')[0]` remains — it must **exclude** the B2 printer-prefix + B3 shelf-label + C scan-token families (or it false-fails). Target: "no `derive_parent_id_from_prefix`, no `parent_id` derived from a prefix."
- **Python 3.9 runtime** — the new `location_prefix()` helper + `save_locations_list` validation must avoid 3.10+ syntax.

## Risks to respect
- **Blast-radius regression (the big one).** The 4 `spoolman_api` string matchers back room-level clear/delete scope + the fail-closed printer_map-removal guard. Changing them from flat to transitive would let a room clear unmount an actively-printing toolhead spool — invisibly. **Never swap them to `is_descendant`; re-point onto the explicit flat helper.** Review with the [[feedback_adversarial_review_runtime_lens]].
- **Migration idempotency on unmigrated installs.** Deleting the derive function without reworking the `old_default` guards ([:1058](../../../inventory-hub/locations_db.py#L1058), [:1284](../../../inventory-hub/locations_db.py#L1284)) breaks boot-time convergence on any install not yet at fixpoint — and migrations swallow exceptions, so it fails *silently*. Prove no-op or re-express first.
- **Write-check false-rejects.** The `save_locations_list` reject runs against the migrations' own mid-sequence saves. It must permit `parent_id=None` + pseudo-prefixes, and ideally validate only the final persisted shape, or it silently blocks schema convergence.
- **CI canaries + the whole Phase-2 suite go red on purpose.** Update them deliberately (don't just delete) or Phase-5 loses its regression coverage for the room-derivation consumers.
- **`locations.json` is a directory bind-mount** (not the `config.json` single-file EBUSY case) — `save_locations_list`'s `os.replace` is safe; don't route it through the config-style in-place fallback ([[reference_fcc_config_json_bind_mount]], [[project_group32_release_2026_07_05]]).
