# Sub-Location Add Redesign — Phased Plan & Design Decisions

> **Status:** SCOPING (2026-07-07). Research complete (6-agent grounded workflow, all claims file:line-grounded — shared with the L271 Phase-5 scope). No code yet — this doc + the design decisions below gate the build.
> **Feature:** "Adding new locations, specifically shelve, row section locations is messy… any location that's a sub location isn't easy to add. This whole workflow needs to be re-designed." (Feature-Buglist.md line 27.)
> **Part of the 🌳 Location-tree cluster** — see [planning-sprint-epic-grouping.md](planning-sprint-epic-grouping.md). Tightly coupled to [L271-phase5-prefix-retirement-plan.md](L271-phase5-prefix-retirement-plan.md); the two should be built as one working group (commonality pass below).

## TL;DR — the data model is done; this is a UX-layer redesign

The hard part already shipped. The tree is rendered **solely from `parent_id`** (a recursive DFS in `_renderLocationsPayload`, zero dash-parsing — [inv_core.js:593](../../../inventory-hub/static/js/modules/inv_core.js#L593)), the Edit modal already has an **explicit, cycle-safe Parent selector + breadcrumb** ([inv_loc_mgr.js:1668](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1668)), and the write path already validates an explicit `parent_id` ([routes_locations.py:324](../../../inventory-hub/routes_locations.py#L324)). Labels are **not** coupled to the hierarchy — location QR is opaque `LOC:<id>`, scan is a straight lookup, no segment parsing ([labels_csv.py:339](../../../inventory-hub/labels_csv.py#L339), [routes_scan.py:749](../../../inventory-hub/routes_scan.py#L749)).

**What's "messy" is the ADD flow.** To create a nested location today the user fights three redundant, hand-managed inputs that all re-encode the same fact:

1. **One global blank modal, no context.** The only add entry is a global "➕ Add Location" button ([modals_loc_mgr.html:31](../../../inventory-hub/templates/components/modals_loc_mgr.html#L31)) → `openAddModal` opens a fully blank form ([inv_loc_mgr.js:1790](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1790)). There is **no per-row "➕ Add child under THIS node"** affordance — the tree rows only carry QR / Edit / Delete / Manage ([inv_core.js:761](../../../inventory-hub/static/js/modules/inv_core.js#L761)). You lose your place in the tree the moment you click Add.
2. **Hand-typed free-text LocationID.** Plain `<input>`, placeholder "e.g., BOX-1", never generated, never uppercased, never format-checked ([modals_loc_mgr.html:64](../../../inventory-hub/templates/components/modals_loc_mgr.html#L64), [inv_loc_mgr.js:1757](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1757)). When Parent = "Auto" (the default), the parent is derived by splitting this ID on `-` ([inv_loc_mgr.js:1539](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1539)) — so the ID's dash-prefix and the explicit-Parent pick can silently disagree, and the explicit pick wins while the "human-readable" ID no longer matches the tree ([inv_loc_mgr.js:1764](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1764)).
3. **Flat, kind-grouped Parent `<select>`.** `_populateParentSelect` groups every location by *Type* into `<optgroup>`s (Room / Wall Shelf / Row / …), **not by ancestry** — to nest deep you scroll a flat kind-bucket and recognize the row by name; the breadcrumb only confirms *after* you pick ([inv_loc_mgr.js:1690](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1690)).
4. **No inference + a silent footgun.** Type is picked entirely separately from Parent (a child of a Row is *not* defaulted to Section — [inv_loc_mgr.js:1594](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1594)). Max Spools defaults to **1**, which is wrong for the structural grouping rows (Wall Shelf / Row / Section use **0** — [modals_loc_mgr.html:91](../../../inventory-hub/templates/components/modals_loc_mgr.html#L91) vs [locations_db.py:1239](../../../inventory-hub/locations_db.py#L1239)).
5. **Intermediate levels can only be born from a boot migration.** Wall Shelf + Row grouping rows are synthesized *only* by `migrate_shelf_grouping_rows_if_needed` parsing `ROOM-WALL-ROW-SECTION` labels at startup ([locations_db.py:1118](../../../inventory-hub/locations_db.py#L1118)). There is no UI to explicitly create "a new Row under this Wall Shelf" — you have to name a Section with the right dashes and hope the next boot nests it.

So the redesign inverts today's fragile model — *"parent is implied by the typed ID"* — into *"the ID is implied by the picked parent."* That's exactly the direction L271 Phase-5 wants (make `parent_id` authoritative), which is why the two are one working group.

## Design decisions (the forks) — ✅ DECIDED 2026-07-07 (Derek: "Full redesign")

Derek chose the **full redesign** — every S-fork resolves to its recommended option: **S1 = both entry points · S2 = auto-generate the id (editable) · S3 = tree picker via a "Browse tree…" affordance · S4 = infer Type + Max Spools · S5 = explicit grouping-row creation.** Table retained below for the rationale. **Sequencing (Q4): Phase-5 prefix-retirement lands FIRST**, so this flow is built on an authoritative explicit-`parent_id` model.

> **🚫 Bound by the cluster's NO-FORCED-RELABELING invariant** (Derek 2026-07-07 — see [L271-phase5 plan](L271-phase5-prefix-retirement-plan.md#-hard-invariant--no-forced-relabeling-derek-2026-07-07)). **S2 auto-generation applies to NEW rows ONLY** — it must never rename or re-mint an existing LocationID (that would orphan a physical label and add to Derek's relabeling backlog). Existing ids stay exactly as printed; a rename remains a deliberate, opt-in user action that is the *only* thing allowed to trigger a (batchable, non-nagging) relabel.

| # | Fork | Options | **Decision** / Recommendation |
|---|------|---------|----------------|
| S1 | **Per-node "Add child" entry point** | (a) per-row "➕ Add child" that pre-seeds Parent=this row (+ inferred Type/ID/Max) · (b) keep the single global button, just improve the picker · (c) both | **(c) both** — per-row add-child is the single biggest fix for "messy" (keeps tree context); global button stays for top-level. Mirrors `openEdit`'s pre-population. |
| S2 | **Auto-generate the LocationID** | (a) suggest a dashed ID from parent + type + next sibling index, editable override · (b) keep hand-typed free text · (c) auto only when launched via Add-child (parent known) | **(a/c)** — auto-derive the ID *from* the picked parent (inverts today's fragile "parent implied by ID"). **Decisions with a safe default (stated, not blocking — flag if you disagree):**
- **Keep the existing save contract** (`__auto__` → omit `parent_id`; `__none__` → explicit `null`; a row → that LocationID — [inv_loc_mgr.js:1762](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1762)). A picker swap that stops emitting the sentinels would silently re-stamp `parent_id` on every field edit (the exact regression the Phase-3.5 review-fix #4 preserve-guard exists to prevent). **The redesign changes how the value is *chosen*, never how it's *sent*.**
- **The tree widget owns its own `/api/locations` fetch** rather than assuming `state.allLocations` is warm (it's only populated as a side-effect of the LM table render — [inv_core.js:640](../../../inventory-hub/static/js/modules/inv_core.js#L640) — so a picker opened cold would be empty/stale).
- **Preserve the self+descendant cycle guard** exactly ([inv_loc_mgr.js:1673](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1673)) and keep it in sync with the backend `is_descendant` ([locations_db.py:430](../../../inventory-hub/locations_db.py#L430)).
- **Don't touch the label CSV columns / `LOC:` QR grammar** — they're a pinned contract (test_label_csv_export + the physical `.lbx` P-touch DB link). Arbitrary LocationIDs (incl. spaces / mixed case) already round-trip ([labels_csv.py:550](../../../inventory-hub/labels_csv.py#L550)); new validation must not retroactively reject existing valid-but-unconventional ids.

## ⚠️ Adversarial-review corrections (2026-07-07)

1. **Scope the per-row "➕ Add child" to shelf/leaf kinds — SUPPRESS it on Printer / Tool Head / MMU Slot rows (HIGH).** Those rows' source of truth is the **printer-map editor** (`routes_bindings.py` → `migrate_printers_to_rows_if_needed` / `migrate_printer_map_to_toolheads_if_needed`; the active map is rebuilt SOLELY from Printer rows' `toolheads[]` — [locations_db.py:795](../../../inventory-hub/locations_db.py#L795)). A toolhead created via `api_save_location` would be a tree row **invisible to Quick-Swap + print-deduct** — a source-of-truth split. So: Add-child appears on Room / Wall Shelf / Row / Section / Cart / Storage / Dryer Box; on a Printer/toolhead row it's absent (or routes to the printer-map editor). S2 auto-id + S4 inference are defined for the shelf/leaf hierarchy only; do **not** silently extend them to printer topology. *(Alternative if Derek wants LM-created toolheads: define the full parent-kind→child-kind + id map AND reconcile with `routes_bindings` so the new toolhead lands in `printer_map` — larger, not recommended for v1.)*
2. **S4 inference safe default (LOW).** When the parent's kind is unknown (a custom `localStorage` type or any Type not in the shelf map), fall through to the current `openAddModal` defaults (Type `Storage`, Max Spools `1` — [inv_loc_mgr.js:1795](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1795)), **never** the "structural → Max 0" branch (that would make the child non-loadable). Inference is a non-binding pre-fill the user always sees and can change.
3. **NO-RELABEL invariant: ground it on the `LOC:` label prefix, not "bare ids resolve" (LOW).** The load-bearing guarantee is that printed location labels carry `LOC:{id}` ([labels_csv.py:342](../../../inventory-hub/labels_csv.py#L342)) which resolves via a straight uppercased lookup ([logic.py:209-213](../../../inventory-hub/logic.py#L209)). Bare prefix-less scan resolution is only *conditional* (numeric-only ids resolve as spool/filament; ≤2-char ids error — [logic.py:316](../../../inventory-hub/logic.py#L316)/327), so don't lean on it. Corollary for S2: the **auto-generated id must stay non-numeric + `LOC:`-labeled** (the dashed `XX-YY-ZZ-NN` form already satisfies this — just never emit an all-numeric id).
4. **Tree-picker (S3) writes back into the existing `<select>` (already the decided hybrid) — spell out the no-node case (LOW).** For a row whose current `parent_id` has no selectable node (a PM/PJ/TST pseudo-prefix, or a Virtual/excluded-Type parent), the browse overlay must default to and leave the field at the `__auto__` sentinel (omit `parent_id` → backend preserves), exactly as `_populateParentSelect` falls back today ([inv_loc_mgr.js:1707](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1707)). The hybrid design already preserves the save contract structurally; just don't let the overlay force a concrete node onto such rows. Pin with a "edit Name on a PM/PJ-parented row → `parent_id` unchanged" test.

## Phased build plan

- **Phase 1 — Frictionless add-child + inference (S1/S4).** A per-row "➕ Add child" button in the tree ([inv_core.js:761](../../../inventory-hub/static/js/modules/inv_core.js#L761) row-action cell + the delegated handler wiring) that opens `#locModal` pre-seeded with Parent = the clicked row, Type inferred from the parent's kind, and Max Spools inferred (0 for structural, sensible default otherwise; **unknown parent kind → `Storage`/`1`, never structural/0** — correction #2). Mirror `openEdit`'s pre-population. **Scope the button to shelf/leaf kinds — SUPPRESS on Printer / Tool Head / MMU Slot rows** (correction #1; those go through the printer-map editor, else you split the toolhead source of truth). Fixes the "lost context" + footgun complaints with the smallest surface.
- **Phase 2 — Auto-generated LocationID (S2).** Given the chosen parent + type + next sibling index, propose an editable `XX-YY-ZZ-NN` id (client-side, using the same segment vocabulary the shelf migration understands). Removes the last hand-typed hierarchy input. **Depends on Phase-5's explicit-parent model** so the generated id and the explicit parent can't be independently reconciled by the still-live Auto path (see sequencing risk below).
- **Phase 3 — Tree picker widget (S3).** A `mountOverlay` expand/collapse tree built over the shared `buildLocationTree(rows)` helper (**extracted once in the group's Phase 0** — this phase CONSUMES it, doesn't re-extract; see the commonality pass), reachable via "Browse tree…" from the edit modal. Emits the same `parent_id` save contract (correction #4 — the no-node/pseudo-parent case stays `__auto__`). Keyboard-nav per the `.kb-active` idiom.
- **Phase 4 — Explicit grouping-row creation (S5).** "Create missing levels" when adding under a room that has no Wall Shelf/Row yet; demote `migrate_shelf_grouping_rows_if_needed` to a legacy-only backfill (leave it running for old installs, but new nesting no longer needs it).
- **Phase 5 — Polish + tests.** Visual baseline for the new add flow; unit/E2E pins for add-child pre-seed, id-generation, inference defaults, cycle rejection; confirm the label CSV + QR grammar are unchanged.

## Risks to respect
- **Build-on-sand if sequenced wrong.** Building the nicer add flow on top of the still-live Auto-derive path (`immediate_parent_for` / `derive_parent_id_from_prefix`) re-introduces the ID-prefix-vs-explicit-parent silent mismatch. **Do Phase-5's explicit-parent hardening first** (see sequencing fork in the cluster doc).
- **Save-contract regression.** The Auto/None/explicit sentinel semantics are entangled with backend `parent_id` persistence ([inv_loc_mgr.js:1762](../../../inventory-hub/static/js/modules/inv_loc_mgr.js#L1762) ↔ [routes_locations.py:360](../../../inventory-hub/routes_locations.py#L360)). Preserve them.
- **Cold-fetch picker.** `state.allLocations` is only warm after a LM table render — the widget must self-fetch.
- **Shelf-migration is a prod data migration, not a refactor** — it runs every boot and is idempotent-by-guard; any change risks re-nesting/re-typing prod rows on the next deploy ([locations_db.py:1118](../../../inventory-hub/locations_db.py#L1118)).
- **No write-time orphan guard exists yet** — orphan tolerance is render-time only (an orphan floats to root, [inv_core.js:596](../../../inventory-hub/static/js/modules/inv_core.js#L596)). If the redesign starts creating deep chains, Phase-5's write-time `parent_id` validation must land with it or silently-orphaned rows will "work" only because the renderer floats them.
- **Relabeling backlog (Derek's constraint).** Auto-generated ids must apply to NEW rows only; never regenerate an existing id. Any UI copy about ids/labels should not imply the user must reprint existing labels. Verify the scan resolver still lands bare/`LOC:`/legacy location labels (see the Phase-5 invariant's edge-case-checklist item) before this ships.

## Prior art / alignment
The Phase-5 Edit-UI slice (2026-06-05, `main`) is the direct predecessor — it shipped the explicit Parent selector, breadcrumb, data-driven Type dropdown, and the shelf-nesting migration. This epic is the *next* layer: turn those primitives into a guided add flow. Align the tree-picker widget with the **shared `buildLocationTree` helper** the commonality pass proposes (the same child-map the LM table renders from), so the picker and the manager always show identical structure.
